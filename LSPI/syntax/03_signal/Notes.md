# 1. Signal Fundamentals

## What is a Signal?

A signal is a software interrupt generated by the kernel or another process.

The kernel may suspend a process at almost any instruction, invoke a signal handler, and then resume execution after the handler returns.

## Signal Lifecycle

```text
Generated
    ↓
Pending (if blocked)
    ↓
Delivered (handler executes)
```

## Standard Signals Are Not Queued

For standard signals (typically 1–31):

- Pending status is represented by a bit mask.
- Multiple arrivals of the same signal collapse into a single pending instance.

Example:

```text
SIGUSR1 arrives 10 times while blocked
            ↓
Kernel records only one pending SIGUSR1
            ↓
Handler executes once after unblocking
```

# 2. Signal Sets and Signal Masks

Linux uses `sigset_t` to represent a collection of signals.

## Signal Set Operations

### Initialize an Empty Set

```c
sigemptyset(&set);
```

### Initialize a Full Set

```c
sigfillset(&set);
```

### Add a Signal

```c
sigaddset(&set, SIGINT);
```

### Remove a Signal

```c
sigdelset(&set, SIGINT);
```

### Test Membership

```c
sigismember(&set, SIGINT);
```

## Blocking and Unblocking Signals

```c
int sigprocmask(
    int how,
    const sigset_t *set,
    sigset_t *oldset);
```

### SIG_BLOCK

Add signals to the current blocked set.

```c
sigprocmask(SIG_BLOCK, &set, NULL);
```

### SIG_UNBLOCK

Remove signals from the blocked set.

```c
sigprocmask(SIG_UNBLOCK, &set, NULL);
```

### SIG_SETMASK

Replace the current mask entirely.

```c
sigprocmask(SIG_SETMASK, &set, NULL);
```

## Signals That Cannot Be Blocked

```text
SIGKILL
SIGSTOP
```

# 3. Async-Signal-Safe Handler Design

Signal handlers execute asynchronously and may interrupt code at arbitrary points.

## Use Only Async-Signal-Safe Functions

Safe examples:

```c
write()
_exit()
```

Unsafe examples:

```c
printf()
malloc()
free()
exit()
```

These functions use internal buffers and locks and may cause deadlocks or undefined behavior.

## Use `volatile sig_atomic_t`

```c
volatile sig_atomic_t flag = 0;
```

Benefits:

- Prevents compiler optimization issues.
- Guarantees atomic reads and writes.

Avoid:

```c
flag++;
flag--;
```

These operations may not be atomic.

## Preserve errno

```c
void handler(int sig)
{
    int saved_errno = errno;

    /* handler code */

    errno = saved_errno;
}
```

# 4. Advanced Signal Configuration with sigaction()

`sigaction()` is the recommended POSIX API for signal handling.

## Extended Information: SA_SIGINFO

Enable:

```c
sa.sa_flags = SA_SIGINFO;
```

Handler:

```c
void handler(
    int sig,
    siginfo_t *info,
    void *ucontext);
```

Useful fields:

```c
info->si_pid
info->si_uid
info->si_code
info->si_addr
```

Examples:

```text
FPE_INTDIV
SEGV_MAPERR
```

## Automatic Restart: SA_RESTART

Enable:

```c
sa.sa_flags |= SA_RESTART;
```

Automatically restarts many interrupted system calls:

```c
read()
write()
accept()
```

Exceptions:

```c
select()
poll()
epoll_wait()
sleep()
nanosleep()
```

# 5. Useful Signal Functions

## kill()

```c
kill(pid, sig);
```

### Null Signal

```c
kill(pid, 0);
```

Results:

```text
0      -> Process exists
EPERM  -> Exists but permission denied
ESRCH  -> Process does not exist
```

## raise()

```c
raise(SIGUSR1);
```

Sends a signal to the calling thread/process.

## sigsetjmp() / siglongjmp()

Signal-aware versions of:

```c
setjmp()
longjmp()
```

Restore the saved signal mask when returning.

## sigaltstack()

Creates an alternate signal stack.

Combined with:

```c
SA_ONSTACK
```

Useful for handling stack-overflow-generated SIGSEGV.

## pause()

```c
pause();
```

Sleeps until a signal arrives.

# 6. Core Dump Files

## What Is a Core Dump?

A snapshot of a process's memory image at the moment it terminates due to a fatal signal.

Examples:

```text
SIGSEGV
SIGABRT
SIGQUIT
```

Purpose:

- Post-mortem debugging
- GDB analysis
- Inspecting variables and call stacks

## Creating Core Dumps

Keyboard:

```bash
Ctrl + \
```

Generate:

```text
SIGQUIT
```

Using GDB:

```bash
gcore <pid>
```

## Reasons Core Dumps May Not Be Generated

### Permission Problems

- Cannot write current directory
- Existing core file is not writable

### Resource Limits

```bash
ulimit -c
```

Enable:

```bash
ulimit -c unlimited
```

### Security Restrictions

- Set-UID processes
- Set-GID processes

### Filesystem Problems

- Disk full
- No free inodes
- Quota exceeded
- Read-only filesystem

## Linux-Specific Configuration

```text
/proc/sys/fs/suid_dumpable
/proc/PID/coredump_filter
/proc/sys/kernel/core_pattern
```

Common specifiers:

```text
%p -> PID
%e -> Executable name
%t -> Timestamp
```

# 7. Special Signal Behaviors

## SIGKILL and SIGSTOP

These signals:

```text
Cannot be caught
Cannot be ignored
Cannot be blocked
```

## SIGCONT and Stop Signals

Stop signals:

```text
SIGSTOP
SIGTSTP
SIGTTIN
SIGTTOU
```

Resume signal:

```text
SIGCONT
```

Rules:

- SIGCONT always resumes a stopped process.
- Delivering SIGCONT discards pending stop signals.
- Delivering a stop signal discards pending SIGCONT.

## Terminal Signal Convention

Programs should respect inherited ignored dispositions for:

```text
SIGHUP
SIGINT
SIGQUIT
SIGTSTP
SIGTTIN
SIGTTOU
```

# 8. Process Sleep States

| State | ps | Signal Response |
|---------|---------|---------|
| TASK_INTERRUPTIBLE | S | Wakes immediately |
| TASK_UNINTERRUPTIBLE | D | Ignores signals |
| TASK_KILLABLE | - | Responds to fatal signals |

## TASK_INTERRUPTIBLE

Waiting for:

- Terminal input
- Pipe data
- Semaphores

Signals wake the process.

## TASK_UNINTERRUPTIBLE

Waiting for:

- Disk I/O
- NFS operations

Signals remain pending.

Even SIGKILL cannot terminate it.

## TASK_KILLABLE

Like TASK_UNINTERRUPTIBLE but responds to fatal signals.

# 10. Synchronous vs Asynchronous Signals

## Asynchronous Signals

Generated externally:

```text
kill()
Ctrl-C
Child termination
```

Arrival time is unpredictable.

## Synchronous Signals

Generated by the current execution flow:

```text
SIGSEGV
SIGFPE
raise()
kill(getpid(), ...)
```

Predictable and reproducible.

# 11. Signal Delivery Timing and Ordering

## Delivery Timing

Linux checks pending signals when transitioning:

```text
Kernel Mode
    ↓
User Mode
```

Examples:

- End of a system call
- Scheduler dispatch

## Ordering

Linux typically delivers signals in ascending signal-number order.

Example:

```text
SIGINT (2)
before
SIGQUIT (3)
```

POSIX does not guarantee this ordering.

## Nested Handlers

Signal handlers may interrupt other handlers, creating nested execution.

# 12. signal() vs sigaction()

## Problems with Traditional signal()

Historically:

- Handler automatically reset
- No self-blocking
- No automatic restart

Potential issues:

```text
Race conditions
Recursive handlers
Interrupted system calls
```

## Recommendation

Always prefer:

```c
sigaction()
```

Use `signal()` only for simple disposition changes such as:

```c
signal(SIGINT, SIG_IGN);
signal(SIGINT, SIG_DFL);
```

# 13. Realtime Signals

## Advantages

### Queued Delivery

```text
10 sends
↓
10 deliveries
```

### Attached Data

Use:

```c
sigqueue()
```

to send:

```c
int
void *
```

payloads.

### Ordering

Different signals:

```text
Lower signal number
→ Higher priority
```

Same signal:

```text
FIFO
```

## Identifiers

```c
SIGRTMIN + n
```

through

```c
SIGRTMAX
```

## Queue Limits

Controlled by:

```text
RLIMIT_SIGPENDING
```

Queue overflow:

```text
EAGAIN
```

## Sending

```c
union sigval value;
value.sival_int = 123;

sigqueue(pid, SIGRTMIN, value);
```

## Receiving

Requires:

```c
SA_SIGINFO
```

Example:

```c
int data = info->si_value.sival_int;
```

Check:

```c
info->si_code == SI_QUEUE
```

# 14. Advanced Signal Synchronization

## sigsuspend()

Atomically:

1. Changes signal mask
2. Sleeps waiting for a signal

```c
sigsuspend(&mask);
```

Avoids race conditions between:

```c
sigprocmask()
pause()
```

## sigwaitinfo()

```c
sigwaitinfo(&set, &info);
```

Signal must already be blocked.

Consumes the signal directly without invoking a handler.

## sigtimedwait()

Like:

```c
sigwaitinfo()
```

but supports timeouts.

## signalfd()

Linux-specific API:

```c
int fd = signalfd(...);
```

Read signals as normal file data:

```c
read(fd, ...)
```

Can integrate with:

```c
select()
poll()
epoll()
```

# 15. Legacy UNIX Signal APIs

| Task | System V | BSD | POSIX |
|--------|----------|---------|---------|
| Install handler | sigset() | sigvec() | sigaction() |
| Block signal | sighold() | sigblock() | sigprocmask() |
| Unblock signal | sigrelse() | sigsetmask() | sigprocmask() |
| Ignore signal | sigignore() | - | signal(SIG_IGN) |
| Safe wait | sigpause() | sigpause() | sigsuspend() |

## BSD Notes

BSD historically used:

```c
int sv_mask;
```

instead of:

```c
sigset_t
```

and automatically restarted interrupted system calls unless explicitly disabled.
